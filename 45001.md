# Plan de développement — SMS SST / ISO 45001:2018

## 1. Objectif
Ce document constitue le **plan de référence** pour l’extension du module QHSE ENSP vers une conformité démontrable à **ISO 45001:2018**.

Le principe est de conserver les modules déjà réalisés, de les **lier entre eux** par les références communes (processus, risques, documents, acteurs), puis de compléter les blocs manquants.

---

## 2. Socle existant et liens transverses

| Module | État | Clause ISO 45001 | Relations avec les autres modules |
|---|---|---|---|
| **PTW** (Permis de travail) | Réalisé | `§8.1.3` | ← HIRA (risques), ← AEI (risques environnementaux), ← Habilitations (responsable/compétences), ← Incidents (zone/processus) |
| **HIRA** (Dangers / risques) | Réalisé | `§6.1.2` | → PTW (mesures de maîtrise), → AEI (aspects environnementaux), ↔ Incidents (causes/gravité), ← Conformité (textes réglementaires) |
| **AEI** (Aspects environnementaux) | Réalisé | `§6.1.2` / `§9.1.2` | ← HIRA (risques partagés), ← Conformité (veille réglementaire), ← Indicateurs (KPIs environnementaux) |
| **Incidents / AT** | Réalisé | `§10.2` | → HIRA (causes/gravité), ← PTW (zone), ← Habilitations (responsables AT) |
| **Habilitations SST** | Réalisé | `§7.2` | → PTW (responsables), → Incidents (personnes concernées), ← HIRA (compétences requises) |
| **Conformité légale** | Réalisé | `§9.1.2` | → HIRA (dangers réglementés), → AEI (textes environnementaux), → Incidents (obligations déclaratives) |
| **Indicateurs environnementaux** | Réalisé | `§9.1` (ISO 14001) | ← AEI (données brutes), ← Conformité (seuils réglementaires), ← HIRA (risques environnementaux) |

### Champs communs à systématiser
- **Processus** : déjà présent dans HIRA, AEI, PTW ; à ajouter dans Incidents, Habilitations, Conformité.
- **Référence documentaire** : `document_ref` dans HIRA et AEI ; à aligner avec le référentiel documentaire existant.
- **Responsable / acteur** : `responsable` dans PTW, Incidents ; à ajouter dans Habilitations, Conformité.

## 3. Lacunes de conformité ISO 45001

### Bloc 1 — Préparation et réponse aux urgences (`§8.2`)
- **POI / Plans d’intervention** : absence de module dédié.
- **Consignes d’évacuation** : pas de référentiel structuré.
- **Exercices de mise en situation** : pas de suivi.
- **Lien avec Incidents/AT** : stocker les causes/défauts de préparation.
- **Lien avec HIRA** : alimenter les risques « situation d’urgence ».

### Bloc 2 — Participation et consultation (`§5.4`)
- **Représentants SST** : pas de module dédié.
- **Remontées terrain / suggestions** : pas de suivi.
- **Commissions SST / CHSCT** : pas de module de suivi.
- **Lien avec Incidents/AT** : canal de signalement.
- **Lien avec PTW/HIRA** : consultation avant travaux dangereux.

### Bloc 3 — Communication SST (`§7.4`)
- **Briefings sécurité / toolbox** : pas de module.
- **Diffusion consignes** : pas de suivi.
- **Formation continue** : lien avec Habilitations.
- **Lien avec Incidents/AT** : retour d’expérience / sensibilisation.

### Bloc 4 — Objectifs SST et programmes (`§6.2`)
- **Cibles annuelles** : pas de module.
- **Programme de réalisation** : pas de suivi.
- **Responsables / échéances** : lien avec Habilitations.
- **Lien avec Incidents/AT** : indicateurs de performance (TF, TFP).

### Bloc 5 — Surveillance et mesure (`§9.1`)
- **Indicateurs SST** : absence pour le SMS (seuls indicateurs environnementaux existent).
- **Inspections / rondes SST** : pas de module.
- **Audit interne SMS** : pas de module.
- **Lien avec HIRA** : fréquence de contrôle des risques.
- **Lien avec Incidents/AT** : TF, TFP, taux de gravité.

### Bloc 6 — Amélioration continue (`§10.3`)
- **Veille / benchmarking SST** : pas de module dédié.
- **Analyse de tendances** : pas de module.
- **Retour d’expérience** : lien Incidents / PTW / HIRA.

## 4. Plan de développement détaillé

### 4.1 Module Urgences / POI — Priorité 1
**Clause ISO 45001** : `§8.2`
**Dépendance** : HIRA (risques urgences), Incidents/AT (causes), AEI (pollution accidentelle)

#### Actions :
1. Créer la table `poi` (POI, zone, type urgence, consignes, responsables, exercices).
2. Créer la route backend `/api/poi` (CRUD).
3. Créer le composant frontend `POI.tsx` :
   - Liste des POI par zone
   - Détail : consignes, plan d’évacuation, numéros d’appel
   - Suivi des exercices et tests
4. Lier chaque POI à :
   - un ou plusieurs risques HIRA
   - une zone PTW
   - un type d’incident AT
5. Ajouter dans Incidents/AT un champ `lien_poi` pour tracer la préparation.

### 4.2 Module Objectifs SST — Priorité 2
**Clause ISO 45001** : `§6.2`
**Dépendance** : Habilitations (responsables), Incidents/AT (TF, TFP), Conformité (textes applicables)

#### Actions :
1. Créer la table `objectifs_sst` (objectif, indicateur, cible, responsable, échéance, statut).
2. Créer la route backend `/api/objectifs-sst` (CRUD).
3. Créer le composant frontend `ObjectifsSST.tsx` :
   - Objectifs annuels
   - Indicateurs de suivi
   - Programme de réalisation
4. Lier chaque objectif à :
   - un responsable habilité
   - un ou plusieurs risques HIRA
   - un ou plusieurs incidents AT
5. Afficher dans le tableau de bord la progression globale.

### 4.3 Module Participations / Consultations — Priorité 3
**Clause ISO 45001** : `§5.4`
**Dépendance** : Incidents/AT (signalements), PTW (consultation travaux), Habilitations (acteurs)

#### Actions :
1. Créer la table `participations_sst` (type, date, participants, ordre du jour, décisions).
2. Créer la route backend `/api/participations-sst` (CRUD).
3. Créer le composant frontend `ParticipationsSST.tsx` :
   - Liste des instances (réunions, commissions, consultations)
   - Comptes-rendus et décisions
   - Suivi des actions issues
4. Lier chaque participation à :
   - un ou plusieurs risques HIRA
   - des incidents/AT signalés
   - des agents habilités

### 4.4 Module Audits SST — Priorité 4
**Clause ISO 45001** : `§9.2`
**Dépendance** : Conformité (référentiels), Objectifs SST (programmes), Incidents/AT (non-conformités)

#### Actions :
1. Créer la table `audits_sst` (référence, type, auditeur, date, périmètre, constats, plan d’action).
2. Créer la route backend `/api/audits-sst` (CRUD).
3. Créer le composant frontend `AuditsSST.tsx` :
   - Programme d’audit
   - Constats et classement (NC, observation, point fort)
   - Plan d’action et suivi
4. Lier chaque audit à :
   - un ou plusieurs risques HIRA
   - des objectifs SST
   - des non-conformités / incidents

### 4.5 Module Revue de direction SST — Priorité 5
**Clause ISO 45001** : `§9.3`
**Dépendance** : Objectifs SST, Audits SST, Incidents/AT, Indicateurs

#### Actions :
1. Créer la table `revues_direction_sst` (date, participants, synthèse, décisions, actions à suivre).
2. Créer la route backend `/api/revues-sst` (CRUD).
3. Créer le composant frontend `RevuesSST.tsx` :
   - Synthèse des indicateurs
   - Décisions de direction
   - Suivi des engagements
4. Lier chaque revue à :
   - les objectifs SST en cours
   - les audits programmés
   - les incidents/AT marquants

### 4.6 Module Indicateurs SST — Priorité 6
**Clause ISO 45001** : `§9.1`
**Dépendance** : Incidents/AT (TF, TFP), Habilitations (formations), PTW (permis délivrés), HIRA (risques maîtrisés)

#### Actions :
1. Créer la table `indicateurs_sst` (type, valeur, période, tendance, responsable).
2. Créer la route backend `/api/indicateurs-sst` (CRUD).
3. Créer le composant frontend `IndicateursSST.tsx` :
   - Graphiques TF/TFP/taux de gravité
   - Taux de formation / habilitation
   - Taux de fermeture PTW
   - Taux de réalisation POI
4. Lier chaque indicateur à :
   - un module source (Incidents, Habilitations, PTW, POI)
   - un objectif SST

## 5. Référentiels et données à alimenter

Pour chaque module, les données initiales doivent être cohérentes avec les modules existants :

| Référentiel | Contenu | Lien modules existants |
|---|---|---|
| Processus | Liste officielle des processus | PTW, HIRA, AEI, Incidents, Habilitations, Conformité |
| Risques HIRA | Tous les dangers/risques | PTW, AEI, Incidents, POI, Objectifs SST |
| Aspects AEI | Aspects environnementaux | Conformité, Indicateurs environnementaux |
| Documents applicables | PDF, procédures, modes opératoires | PTW, HIRA, AEI, POI, Objectifs SST |
| Textes légaux | Décrets, arrêtés, normes | Conformité, POI, AEI, Indicateurs environnementaux |
| Agents habilités | Compétences, dates expiration | PTW, Incidents, Audits, Objectifs SST |

## 6. Règles de liaison entre modules

- **Un risque HIRA** peut être lié à :
  - un ou plusieurs PTW (mesures de contrôle)
  - un ou plusieurs POI (urgence)
  - un ou plusieurs incidents/AT (réalisation du risque)
  - un ou plusieurs audits SST (vérification du contrôle)

- **Un incident/AT** peut être lié à :
  - un ou plusieurs risques HIRA
  - un POI (préparation)
  - un objectif SST (correctif)
  - un audit SST (constat)

- **Un PTW** peut être lié à :
  - un ou plusieurs risques HIRA
  - un responsable habilité
  - un POI (situation d’urgence possible)
  - un incident/AT (si défaillance)

- **Un objectif SST** peut être lié à :
  - un ou plusieurs risques HIRA
  - un ou plusieurs incidents/AT
  - un responsable habilité
  - un audit SST (vérification)
